Order Dashboard —
Pending Orders

A daily command center for merchants' order-processing staff — rebuilt to stay fast and clear even for mega-suppliers handling tens of thousands of orders.

Role: Product Manager Rezio (a KKday company) Web · mobile web · app
↓ 50%+ order query time
Context & Problem

A sales dashboard, when staff needed a to-do list

Rezio's original "My Home" dashboard surfaced sales and redemption statistics. But the people who process orders every day needed a different view: what needs manual action right now, and how much of it there is — so they could clear the work before it became a problem.

The pain sharpened at scale. Two years after launch, mega-suppliers were straining the order list:

For high-volume suppliers like Nikko Toshogu, Asahiyama Zoo, HIS and Sky Walker Yachts, a list that took the better part of a minute to load made daily order processing genuinely painful.

Goal

Fast, prioritized, action-first — at any scale

Create a dedicated pending-orders dashboard that lets staff instantly grasp order status, view lists and details by category, and act on orders directly — and that stays responsive even at tens of thousands of orders.

Solution

From "viewing data" to "taking action"

An action-first layout

Three zones — Orders, Registrations and Problem orders — each with a running total, plus 18 quick-filter categories (e.g. past confirmation deadline, needs confirming within a day, voucher to send, overdue unconfirmed). Staff click a category and immediately get the matching orders.

Deadline color flags

Each order card carries a deadline flag in the bottom-right corner, so urgency is visible at a glance without reading a single date:

< 24 hr or overdue — red 3–7 days — yellow > 7 days — blank

Continuous processing

Order details open in an embedded pop-up with a "Next" button, so staff can work through a batch back-to-back without leaving the page. The list keeps the snapshot captured when opened, so the sequence doesn't jump around mid-task.

Performance — defined the requirements, engineering built it

  • Time-window limits per category (e.g. general orders show only from today onward, or those created within the last ~90 days) to shrink the volume of data loaded.
  • Infinite scroll — load the next 20 orders as the user scrolls, instead of fetching everything up front.
  • Optimized list UI and search, which together cut order query time by 50%+.
Highlights

Why it worked

Reframing the dashboard around the job

The biggest move wasn't visual — it was deciding that order-processing staff need an action queue, not a sales report, and designing the whole surface around that single job.

Designing for scale without throwing hardware at it

The performance win came from smart product rules — sensible time windows and paginated loading — that I defined as requirements, rather than simply asking for more infrastructure.

Reducing cognitive load

Color-coded deadlines and the continuous-processing pop-up turn a tedious, error-prone chore into a fast, low-friction flow.
