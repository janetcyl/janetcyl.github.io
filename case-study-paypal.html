Cross-border
Payments — PayPal

Led the PayPal integration so travel-experience merchants could take payments from overseas visitors — by reading the payment API docs end to end and turning them into fail-safe specs.

Role: Product Manager (owner) 2025 Rezio (a KKday company)
↑ 15%+ expected order completion rate
Context & Problem

Local-only payments couldn't serve overseas travelers

Rezio's merchants are mostly small and mid-sized travel-experience operators. As international travel rebounded and cross-border bookings grew, their local-first payment setup couldn't serve overseas visitors:

  • Target markets — the Philippines, Thailand, Indonesia and the rest of Southeast Asia — have different payment habits and currencies, calling for multi-currency checkout and a way for visitors without a PayPal account to pay by card.
  • Cross-border payments involve fees, currency conversion, fee-splitting, refund windows and disputes — miss any one and you get dropped orders or books that don't reconcile.
  • Rezio's own data: adding online payment lifts the average order-completion rate by 15%+ — payment convenience is the conversion lever.
Goal

Make PayPal work end to end

Integrate PayPal into Rezio to deliver:

  • Multi-currency checkout (USD, JPY, TWD, PHP, THB, SGD and more — 13 currencies).
  • Pay with a PayPal account, or by credit card directly (no account needed).
  • A complete, traceable fee-split, refund and reconciliation mechanism.
  • A fail-safe flow that doesn't drop orders even when the gateway disconnects or responses lag.
Solution

A non-technical PM who reads the API

Reading the docs, writing the spec — the heart of this project

I'm not from a technical background, yet I read PayPal's official API documentation directly — REST API v2, Multiparty Payments, the Partner Referral API and Order API v2 — and independently listed the send/receive parameters for each call, turning them into specs engineers could build from straight away:

Order API v2 → create order · purchase_units:
  category = "DIGITAL_GOODS"   quantity = 1   …

Making the fee-split logic clear

I mapped and documented the whole money trail — from gross_amount → platform_fees → paypal_fee → net_amount → exchange_rate → receivable_amount — and defined the difference between "collect everything then convert to the primary currency" versus "receive in that currency without conversion," so merchants understand exactly how each payout is calculated.

Designing fail-safes for the edge cases

  • Reconciliation / re-post mechanism: Sandbox negative testing revealed that an HTTP 201 with status PENDING must be treated as not complete — wait for the Webhook, then write the order — so success is never mis-read and no order is dropped.
  • Error-code translation: mapped PayPal's error codes into human-readable prompts with default failure copy.
  • Passing tracking context: used SetTransactionContext to send TRACKING_ID, payer email and session date for reconciliation — and, recognizing that buyers checkout without registering, deliberately omitted the member-account fields.

Rigorous refund & reconciliation rules

  • Refunds: 0–179 days from the transaction date, straight from the Rezio back office, with no extra fee (the original fee isn't returned); past 180 days, disputes go through PayPal's Resolution Center.
  • Reconciliation: payer email and an itemized fee breakdown appear in the order's billing record, so merchants can match it against their PayPal back office.
